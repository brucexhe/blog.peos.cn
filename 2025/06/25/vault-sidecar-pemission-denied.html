✅ 问题定位

❗️核心问题

Vault Agent Injector webhook 依赖 TLS 证书通信（Admission Webhook 机制）。这条错误日志表示它的 TLS 配置有问题，可能来源于：

原因 说明
❌ injector 使用的 TLS 证书过期/不可信 手动或自动 TLS 证书未正确配置或未更新
❌ 未使用官方 helm install 自动管理 webhook TLS 自签证书未被 API Server 信任
❌ 没正确设置 vault-injector 的 mutatingwebhookconfiguration 包含的 CA Bundle 无效或空

✅ 解决方案建议

方式 1：使用 Helm 自动安装修复 TLS

如果你是用 Helm 安装 Vault，可以通过 Helm 来修复 TLS 问题。

bash
helm upgrade --install vault hashicorp/vault \
--namespace vault \
--set "injector.enabled=true" \
--set "injector.tls.generate=true"

  • --set injector.tls.generate=true：自动生成并注册可信 CA，避免 TLS 失败。

  • 该方式会自动更新 webhook 中的 caBundle 字段。

建议：这是最稳妥、推荐的方式，能避免你手动维护 TLS。

方式 2：手动排查 & 更新 webhook CA（高级）

如果你是手动部署的 Vault Injector（非 Helm），你要：

  1. 查看当前 webhook 的配置：

bash
kubectl get mutatingwebhookconfiguration vault-agent-injector-cfg -o yaml

  1. 检查其中的：

yaml
clientConfig:
caBundle: <必须是 base64 编码的 CA>

如果这个字段是空的或无效，K8s API Server 就无法信任 injector 提供的证书，导致 TLS 握手失败。

  1. 你需要手动生成一个 CA，并更新 webhook 配置，参考：官方文档：Manually Managing TLS

✅ 验证 TLS 是否修复成功

一旦你修复 TLS，再重新创建 pod，比如重新 apply test01 deployment，然后：

bash
kubectl describe pod <test01-pod-name> -n test01

你应该能看到：

  • 有注入的 init container 或 vault-agent sidecar container。

  • Pod 的 volume 中挂载了 /vault/secrets/。

✅ 1. 使用 Vault Agent Sidecar 而不是纯模板注入

你已经启用了 agent-injection，但需要配置它为 自动模板刷新模式。

核心参数是：

yaml
vault.hashicorp.com/agent-inject-template-config.json: |
{{- with secret "lululemon/data/aiportal" -}}
{{ .Data.data | toJSON }}
{{- end }}

要加上 template_config 块设置 change_mode 为 signal 或 restart：

yaml
vault.hashicorp.com/agent-inject-template-config.json: |
{{- with secret "lululemon/data/aiportal" -}}
{{ .Data.data | toJSON }}
{{- end }}
{{- /* template_config */ -}}
{{- with secret "lululemon/data/aiportal" | template_config -}}
change_mode = "signal"
change_signal = "SIGHUP"
{{- end }}

或者直接在 annotations 中添加：

yaml
vault.hashicorp.com/agent-inject-template-config.json: |
{{- with secret "lululemon/data/aiportal" -}}
{{ .Data.data | toJSON }}
{{- end }}
vault.hashicorp.com/agent-inject-template-config-change-mode: "signal"
vault.hashicorp.com/agent-inject-template-config-change-signal: "SIGHUP"

✅ 2. 启用 Vault Agent 自动轮询（默认开启）

Vault Agent 会后台轮询 token，有效时刷新 secrets。

但你注入的 secret 文件是否更新，取决于：

  • Vault 的 KV 使用的是 KV v2（你用的是 lululemon/data/...，确认为 v2）

  • 注入配置是否启用了监听（change_mode）

注意事项

  • 如果你想让主容器也对 secret 更新有感知，你需要让它监听 SIGHUP 信号，或文件变化。

  • 例如应用每次读取 secret 时从 /vault/secrets/config.json 中读取，而不是缓存进内存。

  • 或者配合一个 volume + 文件 watcher（比如 sidecar reloaders）。

Vault Agent 默认的 secret **刷新周期（token renew + secret re-rendering）**不是固定时间，而是由以下两个机制决定：

1. Token 自动续期周期

Vault Agent 会自动续期从 Vault 获取的 token，默认行为：

  • 续期时间 ≈ 租期（TTL）的一半

  • 如果你创建的 token 有 60 分钟 TTL，Agent 大约每 30 分钟会尝试续期一次

续期时，也会触发 重新拉取 secret + 渲染 template

2. KV secret 的自动轮询间隔

对于 KV v2 secret，Vault Agent 会使用 long polling（长轮询）机制，由 cache 配置控制。

在 Vault Agent 的内部配置中，相关刷新设置为：

hcl
cache {
use_auto_auth_token = true
}
listener "unix" {
...
}

在没有特别配置的情况下：

 默认刷新周期约为每 5 秒检测一次 secret 变更

不过这个并非在 annotation 中可控，属于 Vault Agent 的默认内部行为，不太容易改。
